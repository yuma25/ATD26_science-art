# APIおよびサービス仕様書

### システムの相互運用性とセキュリティ設計

本アプリケーションは、API レイヤーを介して高度なセキュリティ制御とトラフィック管理を実装しています。

---

## 1. 認証と認可 (Authentication & Authorization)

### 認証方式

- **Supabase Auth (匿名認証)**: ユーザー登録なしで一意のユーザーIDを発行します。デバイスのローカルストレージにセッションを保持し、継続的な体験を提供します。
- **JWT (JSON Web Token)**: クライアントとサーバー間の通信において、Supabase が発行する JWT を使用してリクエストの正当性を検証します。

### 認可 (RBAC)

- **Service Role**: サーバーサイド（API Routes）では、バイパス権限を持つ `SUPABASE_SERVICE_ROLE_KEY` を使用し、管理者レベルの操作（標本獲得の記録等）を実行します。
- **RLS (Row Level Security)**: データベースレベルでポリシーを設定し、ユーザーが自分自身のデータのみを操作できるように制御しています。

---

## 2. セッションとクッキー管理

- **Next.js セッション**: API ルートおよびサーバーコンポーネントにおいて、認証状態を維持するためにセッション管理を行います。
- **Cookie**: Supabase の認証セッションはブラウザの Cookie に安全に保存され、CSRF 対策（SameSite属性等）が施されています。

---

## 3. トラフィック制御とセキュリティ

### レートリミティング (Rate Limiting)

- **API レベル**: 過剰なリクエスト（標本の連続獲得試行など）を防止するため、IP アドレスベースおよびユーザーIDベースのレート制限を適用します。
- **Supabase レベル**: プラットフォーム標準の防御機能により、DDoS 攻撃やブルートフォース攻撃から保護されています。

### CORS (Cross-Origin Resource Sharing)

- 本アプリケーションの API は、同一オリジンからのリクエストのみを許可するように制限されており、外部ドメインからの不正な API 利用を防止しています。

---

## 4. API エンドポイント (抜粋)

### `POST /api/v1/badges/acquire`

- **冪等性設計**: 重複リクエストを許容し、DB のユニーク制約エラーを 200 OK としてハンドリングします。

---

## 5. エラーコード一覧

| コード              | HTTP ステータス | 定義                 |
| :------------------ | :-------------- | :------------------- |
| `INVALID_REQUEST`   | 400             | バリデーション失敗。 |
| `UNAUTHORIZED`      | 401             | 認証エラー。         |
| `TOO_MANY_REQUESTS` | 429             | レート制限超過。     |
| `SERVER_ERROR`      | 500             | 内部エラー。         |

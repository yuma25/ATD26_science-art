# データベースおよびキャッシング戦略仕様書

### データ永続化と高速化の多層設計

PostgreSQL を主データベースとし、アクセス頻度とデータの鮮度に応じたキャッシング戦略を組み合わせています。

---

## 1. データベース戦略 (PostgreSQL)

### リレーショナル設計の利点

- **ACID整合性**: 標本獲得という「記録」の正確性を担保するため、トランザクションをサポートする RDB を採用。
- **複雑な結合**: ユーザー、バッジ、獲得履歴の3要素を柔軟に結合して分析（統計API等）するために最適化。

---

## 2. キャッシング戦略 (Multi-layer Caching)

パフォーマンスの最大化とデータベース負荷の軽減のため、3つのレイヤーでキャッシュを制御します。

### レイヤー1: クライアントサイドキャッシュ (SWR)

- **手法**: `stale-while-revalidate`
- **目的**: 画面遷移時の即時表示。バックグラウンドで最新データを取得し、変更があれば UI を更新します。

### レイヤー2: エッジ / HTTP キャッシュ

- **手法**: `Cache-Control` ヘッダー
- **目的**: 静的な標本マスターデータ（`badges`）など、変更頻度が極めて低いリソースを CDN やブラウザでキャッシュ。

### レイヤー3: サーバーサイド / インメモリキャッシュ (Redis検討)

- **手法**: Key-Value Store
- **現状**: 現在は Supabase の標準パフォーマンスで十分ですが、アクセス増大時には API レイヤーと DB の間に Redis を配置し、頻繁に参照される「全体統計」や「人気標本ランキング」をキャッシュする設計拡張が可能です。

---

## 3. データベース接続と運用

### 接続プーリング

- **PgBouncer**: サーバーレス環境（Next.js API Routes）特有の「大量の短寿命接続」をプーリングし、DBの同時接続数制限を回避します。

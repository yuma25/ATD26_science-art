# データベース設計仕様書

### システムのデータ整合性と永続性

すべてのデータは Supabase (PostgreSQL) によって管理され、タイムスタンプは UTC で処理されます。本設計は、リトライロジックや同時実行リクエスト時においてもデータの一貫性を確保するため、冪等性（Idempotency）に重点を置いています。

---

## 1. テーブル: `badges`

AR体験で使用されるデジタル標本のマスターデータです。

| カラム名       | 型        | Zod バリデーション  | 説明                             |
| :------------- | :-------- | :------------------ | :------------------------------- |
| `id`           | UUID (PK) | `uuid()`            | バッジの一意の識別子             |
| `name`         | String    | `min(1)`            | バッジの表示名                   |
| `description`  | Text      | `string()`          | 標本の詳細な解説                 |
| `target_index` | Integer   | `int().min(0)`      | ARマーカーのインデックス (0-5)   |
| `model_url`    | String    | `url() \|\| path()` | 3Dモデルファイル (.glb) へのパス |

---

## 2. テーブル: `user_badges`

ユーザーによる獲得記録のトランザクションログです。主要な冪等性ロジックを実装します。

| カラム名      | 型        | Zod バリデーション | 説明                         |
| :------------ | :-------- | :----------------- | :--------------------------- |
| `user_id`     | UUID (FK) | `string()`         | profiles.id への参照         |
| `badge_id`    | UUID (FK) | `uuid()`           | badges.id への参照           |
| `acquired_at` | Timestamp | `datetime()`       | 獲得時のタイムスタンプ (UTC) |

### 制約事項

- **複合ユニーク制約 (`unique_user_badge`)**: `(user_id, badge_id)`
  - **目的**: 同一ユーザーによる同一バッジの重複獲得記録を防止します。
  - **動作**: データベースレベルで冪等性を担保し、アプリケーションレイヤーからの安全なリトライを可能にします。

---

## 3. テーブル: `profiles`

ユーザー固有の設定およびステータスです。

| カラム名     | 型        | 説明                                      |
| :----------- | :-------- | :---------------------------------------- |
| `id`         | UUID (PK) | ユーザーID (Supabase Auth.users と紐付け) |
| `party_size` | Integer   | ロジック計算用のパーティー人数            |
| `last_seen`  | Timestamp | 最終アクティビティのタイムスタンプ        |

## 4. クエリとインデックス

### インデックス戦略

- `user_badges.user_id` に対する B-tree インデックス：ユーザーの獲得履歴取得を最適化します。
- `badges.target_index` に対するインデックス：ARマーカー認識時の検索効率を向上させます。

### キャッシングおよび高負荷対策戦略

本システムでは、PostgreSQL の負荷軽減とレスポンス高速化のため、以下の多層キャッシング層を実装しています。

1. **クライアントサイド (SWR)**: `stale-while-revalidate` 戦略により、ブラウザ側でデータを保持しつつバックグラウンドで同期します。
2. **エッジキャッシュ**: 3Dモデル等の静的アセットに対し、CDN による強力なキャッシュを適用しています。
3. **Redis (Upstash) による API キャッシュ (導入済み)**:
   - **対象**: 管理者ダッシュボードの「グローバル統計データ」。
   - **仕組み**: 集計コストの高い統計クエリの結果を Redis に 5分間（300秒）保存します。
   - **効果**: 同時アクセス数が増加しても、PostgreSQL のリソースを消費せずにミリ秒単位のレスポンスを維持します。
